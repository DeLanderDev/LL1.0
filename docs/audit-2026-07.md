# Local Lee — SEO Audit & Security Review

_Date: 2026-07-02. Scope: `server.js`, `public/`, `scripts/deploy.sh` on branch
`claude/seo-audit-security-review-8951vv`._

This document records a point-in-time review of the Local Lee backend
(Node/Express + better-sqlite3) and the static/vanilla-JS frontend. Findings
are grouped **SEO** and **Security**, each ordered by impact. Where a fix was
applied in this branch it is noted inline.

---

## Summary

The site is in good shape. SEO fundamentals are almost all present: unique
titles/descriptions per page, canonicals, Open Graph, JSON-LD, a dynamic
sitemap, `robots.txt`, and an `llms.txt`. The backend is well-built for its
size — parameterized SQL throughout, bcrypt(12) password hashing,
server-side sessions, sensible input length caps, and consistent
client-side HTML escaping. The systemd unit is genuinely hardened.

The issues worth acting on are a small number of SEO gaps on the
JS-rendered detail pages and a handful of security hardening items. None of
the security findings are critical given the app's threat model (a small
community site behind nginx), but two are worth fixing before real traffic:
the missing rate limiting on auth/submission endpoints and the dependency
advisories in `bcrypt`'s build-time chain.

## SEO

### S1 — Detail pages ship generic titles/descriptions in the initial HTML (medium)
`business.html`, `event.html`, `book.html`, `aid-post.html` are static shells
served for every slug. The `<title>`, `<meta name="description">`, canonical,
and OG tags are hardcoded generic ("Business — Local Lee", canonical points to
`/directory`, not the specific item). The real title is only patched in via
JS **after** the API call (`document.title = b.name + …`).

Impact:
- Crawlers that execute JS (Googlebot) will eventually see the updated
  `document.title`, but the **canonical** still points at the list page, which
  tells Google these detail pages are duplicates of `/directory` — they may not
  be indexed on their own.
- The `<meta name="description">` and all `og:`/`twitter:` tags are never
  updated, so social/link previews are generic for every business, event, and
  book.
- Non-JS crawlers and most social scrapers (Slack, Facebook, LinkedIn) don't
  run the fetch, so they only ever see the generic shell.

Recommendation (not auto-applied — needs a rendering decision):
- **Best:** server-render the `<head>` for `/directory/:slug`, `/events/:slug`,
  `/literature/:slug`, `/mutual-aid/post/:id` — inject title, description, and a
  self-referencing canonical from the DB row before sending HTML. The data is
  already loaded server-side for the sitemap, so this is a small change.
- **Minimum:** in each detail page's JS, also set the canonical
  (`link[rel=canonical].href`) and `og:title`/`og:url`/`description` to the
  item's own URL, so JS-capable crawlers stop treating them as duplicates.

### S2 — `aid-post.html` and `book.html` canonicals point at list pages (medium)
Related to S1 but worth calling out: `aid-post.html`'s canonical is
`/mutual-aid` and `book.html`'s is `/literature`. Even with JS, these are never
corrected. If aid posts are meant to be indexable (they expire in 30 days, so
arguably they should be `noindex`), decide explicitly:
- If aid posts should **not** be indexed (they're ephemeral): add
  `<meta name="robots" content="noindex">` to `aid-post.html`. Note they're
  already excluded from the sitemap, which is consistent with noindex.
- If book detail pages **should** be indexed: give them a self-referencing
  canonical (see S1).

### S3 — Sitemap has no `<lastmod>` and omits static submit/auth pages by design (low)
`/sitemap.xml` lists core pages + approved businesses/events/books with
`changefreq weekly` but no `lastmod`. Adding `<lastmod>` from each row's
`created_at` (or an `updated_at` if you add one) is a cheap, legitimate signal.
Login/register/submit pages are correctly excluded. Consider also adding
`/directory/:slug` etc. are present — good.

### S4 — No `og:image` / `twitter:image` anywhere (low)
Every page uses `twitter:card = summary` with no image and no `og:image`. Link
previews will be text-only. Add a single default social image (e.g.
`/img/social.png`, 1200×630) referenced site-wide, and ideally a per-item image
on detail pages later. Low effort, meaningful for shares.

### S5 — Favicon / touch icons missing (low)
No `<link rel="icon">` anywhere. The brand SVG in the header could be exported
to a `favicon.svg` + `.ico` and referenced from the shared `<head>`. Not an SEO
ranking factor but affects brand presentation in tabs and bookmarks.

### S6 — `robots.txt` and `sitemap` hostname are hardcoded to `locallee.org` (low)
`robots.txt` hardcodes `Sitemap: https://locallee.org/sitemap.xml` while the
sitemap loc is built from `SITE_URL`. If the site launches on another domain,
`robots.txt` will point crawlers at the wrong host. Either serve `robots.txt`
dynamically from `SITE_URL` (like the sitemap), or document this in the
production checklist (it partially is). Low risk since the default matches the
intended domain.

**SEO items that are already correct:** unique per-page `<title>` and
`meta description`; one `<h1>` per page; `lang="en"`; viewport meta; canonical
on every page; OG + Twitter card tags; JSON-LD (`WebSite` + `Organization` on
home, `LocalBusiness`/`Event`/`Book` on detail pages); dynamic sitemap; correct
`robots.txt` disallow of `/admin` and `/api/`; `noindex` on the admin page;
semantic HTML with skip-link and `aria-` attributes.

## Security

### SEC1 — No rate limiting on auth or submission endpoints (medium)
`/api/login`, `/api/register`, and all public `POST /api/*` submission routes
have no rate limiting or throttling. This allows:
- Online password brute-forcing against `/api/login` (bcrypt cost 12 slows it,
  but nothing blocks thousands of attempts).
- Spam flooding of the pending queue via `/api/businesses`, `/api/events`,
  `/api/books`, `/api/aid/*` (all accept anonymous submissions).

Recommendation: add `express-rate-limit` (e.g. a strict limiter on
`/api/login` and `/api/register`, a looser one on submission POSTs). nginx
`limit_req` in front is a good defense-in-depth layer too. `trust proxy` is
already set to `1`, so client IPs will be correct behind nginx.

### SEC2 — Build-time dependency advisories in the bcrypt chain (medium)
`npm audit` reports 11 advisories (7 high), almost all transitive through
`bcrypt` → `@mapbox/node-pre-gyp` → `node-gyp`/`tar`/`cacache`, plus a moderate
`qs` DoS via `express@4.21.0`. The tar/node-gyp chain is **build-time only**
(native module compilation), so it is not part of the running attack surface,
but it's still worth clearing:
- `npm audit fix` (non-breaking) resolves the `qs`/express and
  `@tootallnate/once` items.
- Upgrading `bcrypt` to `^6` (breaking major) clears the tar/node-pre-gyp
  chain. `bcrypt@6` is API-compatible for the `hashSync`/`compareSync` calls
  used here; test the native build on the droplet's Node version first.

Recommendation: bump `express` to the latest 4.x and `bcrypt` to `^6` on a
test deploy, re-run `npm ci` on the droplet, and confirm login still works.

### SEC3 — Business claim flow lets any user overwrite a pending/existing owner (low)
`POST /api/businesses/:id/claim` sets `owner_user_id = req.session.userId` and
`claim_status = 'claim_pending'` on **any** business regardless of its current
claim state. A logged-in user can claim a business that's already
`claim_pending` or even `claimed`, overwriting `owner_user_id` and resetting the
status to pending. The admin still has to verify, so it's not a privilege
escalation, but it lets a user knock an existing owner's verification back to
pending. Recommendation: reject the claim if `claim_status` is already
`claimed`, and ideally if it's `claim_pending` by a different user.

### SEC4 — Contact messages are only logged, never delivered (low / functional)
`POST /api/contact` validates input then `console.log`s the message and returns
`ok`. Messages go to the journal only — no email, no DB row. Users believe
they've reached someone. Not a security hole, but worth either wiring to email
or persisting to an admin-visible table so submissions aren't silently lost.
(Also: the logged message includes user-supplied name/email/body in the journal
— low-sensitivity, but be aware it's PII in logs.)

### SEC5 — No explicit Content-Security-Policy (low)
Good headers are set (`X-Content-Type-Options`, `Referrer-Policy`,
`X-Frame-Options`). There is no `Content-Security-Policy`. XSS risk is already
low because all dynamic content is escaped via `LL.escape` before insertion and
JSON-LD is built with `JSON.stringify` (not string concatenation), but a CSP is
strong defense-in-depth. Note the pages use inline `<script>` blocks, so a
strict CSP would require either nonces or moving those scripts to files. A
starter policy: `default-src 'self'; img-src 'self' data:; object-src 'none';
base-uri 'self'; frame-ancestors 'self'`. Add `script-src` handling once inline
scripts are addressed. Consider also `Strict-Transport-Security` once HTTPS is
confirmed (can be set at the nginx layer).

### SEC6 — Session cookie `maxAge` is 30 days with `sameSite: 'lax'` (informational)
Reasonable for a low-risk community site. `httpOnly` and `secure`
(in production) are correctly set. No action required; noted for completeness.
There is no CSRF token, but `sameSite: 'lax'` blocks cross-site POSTs from
forms/fetch, which covers the main CSRF vector for these endpoints. If you later
add cross-origin needs, revisit with a CSRF token.

**Security items that are already correct:** all SQL is parameterized (no string
interpolation of user input into queries — the only interpolated identifiers are
the fixed `APPROVABLE` table map and `uniqueSlug`'s hardcoded table names,
neither user-controlled); passwords hashed with bcrypt cost 12; sessions stored
server-side in SQLite; `x-powered-by` disabled; JSON/body size capped at 256kb;
admin routes gated server-side with `requireAdmin`; all user content escaped
client-side; `rel="noopener"` on outbound links; input lengths capped via
`trim()`; the deploy script and systemd unit are well-hardened
(`NoNewPrivileges`, `ProtectSystem=strict`, dropped capabilities, `PrivateTmp`,
scoped `ReadWritePaths`), and the deploy script refuses to clobber files it
doesn't own and generates a random `SESSION_SECRET`/`ADMIN_PASSWORD`.

## Recommended priority

1. **SEC1** — add rate limiting (login + submissions).
2. **SEC2** — clear dependency advisories (`npm audit fix`; plan bcrypt@6 bump).
3. **S1 / S2** — server-render (or at least JS-correct) detail-page
   `<head>`/canonical so business/event/book pages are individually indexable
   and share cleanly.
4. **SEC3** — tighten the claim flow.
5. **S4 / S5** — add a default social image and favicon.
6. Everything else as time allows.
