# Local Lee — SEO Audit & Security Review

_Date: 2026-07-02. Scope: `server.js`, `public/`, `scripts/build-pages.js`,
`scripts/deploy.sh` on branch `claude/seo-audit-security-review-8951vv`._

This document records a review of the Local Lee backend (Node/Express +
better-sqlite3) and the static/vanilla-JS frontend, **and the fixes applied in
this branch in response to it.** Findings are grouped SEO and Security. Each
carries a status: **FIXED** (changed here), **DEFERRED** (recommended, not yet
done), or **OK** (already correct, no change needed).

> Note on regenerated pages: most `public/*.html` files are generated by
> `scripts/build-pages.js`. Frontend fixes were made in that template and the
> pages regenerated with `node scripts/build-pages.js`, so they won't be lost
> on the next build. `index.html` and `about.html` are hand-authored and were
> edited directly.

---

## Summary

The site was already in good shape — parameterized SQL, bcrypt(12), server-side
sessions, moderation-by-default, a hardened systemd unit, and consistent
client-side escaping on the public pages. The review surfaced **one critical
bug** (stored XSS in the admin dashboard) plus a set of hardening and SEO gaps.
The critical bug and the high-value hardening/SEO items were fixed in this
branch; a few lower-priority items are left as documented recommendations.

---

## Security

### SEC1 — Stored XSS in the admin dashboard — **FIXED** (was critical)
`admin.html`'s `row(label, body, actions)` helper injected `label` into
`innerHTML` **without escaping**, while every caller built `label` from
user-submitted fields that the server only length-trims, never HTML-escapes:
business/event/aid-resource/aid-post `town`, book `author`/`year`, aid
`category`. Those rows render in the queue **while still pending**, so an
anonymous submitter could put `<img src=x onerror=…>` in a `town` field and
run script in the admin's authenticated session on `/admin` — enough to approve
their own listings or exfiltrate the session.

Fix: `label` is now escaped in the `row()` helper
(`<div class="meta">${LL.escape(label)}</div>`) in `scripts/build-pages.js`,
and `admin.html` regenerated. All labels are plain text, so escaping is
complete and safe. (`body`/`actions` were already built from escaped fields.)

### SEC2 — No rate limiting on auth or submission endpoints — **FIXED** (was medium)
`/api/login`, `/api/register`, and all public submission POSTs had no
throttling, allowing password brute-forcing and queue-spam floods.

Fix: added a small in-memory fixed-window rate limiter (`server.js`), keyed by
`req.ip` (correct behind nginx since `trust proxy` is set):
- `authLimiter` — 10 requests / 15 min on `/api/login` and `/api/register`.
- `submitLimiter` — 20 requests / hour on every public submission POST, RSVP,
  book comment, and contact.
A `429` with `Retry-After` is returned when exceeded. Verified: the 9th rapid
login attempt from one IP starts returning `429`. For defense-in-depth, adding
nginx `limit_req` in front is still worth doing later.

### SEC3 — Dependency advisories — **PARTIALLY FIXED** (was medium)
`npm audit` originally reported 11 advisories (7 high). Fixes applied:
- `npm audit fix` cleared the runtime `qs` DoS (via `express`) and the
  `@tootallnate/once` items.
- Bumped `bcrypt` `^5.1.1 → ^6`, which removed the `@mapbox/node-pre-gyp` →
  `node-gyp`/`tar`/`cacache` chain that bcrypt pulled in. bcrypt 6 is
  API-compatible with the `hashSync`/`compareSync` calls used here; the native
  build and login/register were smoke-tested and work.

**Remaining (DEFERRED):** 7 advisories persist, all transitive through
`connect-sqlite3 → sqlite3@5 → node-gyp@8 → tar@6/cacache`. These are
**build-time only** (native compilation), not part of the running attack
surface. Clearing them requires `connect-sqlite3`/`sqlite3` to ship a newer
`node-gyp`, or swapping the session store. Not worth a forced/breaking change
now; re-check when `connect-sqlite3` updates.

### SEC4 — Unsafe `javascript:`/`data:` URLs accepted for websites — **FIXED** (was low/medium)
Submitted `website` fields were stored verbatim and later placed into `href`
attributes. A `javascript:` URL would have been a stored-XSS vector on the
public business/resource pages (escaping the text doesn't neutralize the
scheme). Fix: `normalizeWebsite()` (`server.js`) now accepts only `http(s)`,
prepends `https://` when the scheme is missing, and drops anything with another
scheme. Verified: `javascript:alert(1)` is stored as empty.

### SEC5 — Business claim flow could overwrite an existing owner — **FIXED** (was low)
`POST /api/businesses/:id/claim` set `owner_user_id` on any business regardless
of its claim state, letting a user knock a verified/pending owner back to
pending. Fix: the route now rejects (`409`) a claim on an already-`claimed`
listing, or on one that's `claim_pending` for a different user.

### SEC6 — Contact messages were only logged, never stored — **FIXED** (was low/functional)
`POST /api/contact` did `console.log` and dropped the message (also writing
user PII to the journal). Fix: added a `contact_messages` table; messages are
persisted and now surface in the admin queue (new "Contact messages" section)
with a "Mark handled" action. No longer logged to stdout.

### SEC7 — No Content-Security-Policy — **FIXED** (was low)
Fix: added a CSP header (`server.js`). Because pages use inline `<script>`
blocks, `script-src`/`style-src` include `'unsafe-inline'` for now; the policy
still restricts `default-src`/`connect-src`/`img-src` to `'self'` (+`data:`
images), sets `object-src 'none'`, `base-uri 'self'`, `frame-ancestors 'self'`,
and `form-action 'self'`. Tightening to nonces (dropping `'unsafe-inline'`)
would require moving inline scripts to files — DEFERRED. `Strict-Transport-
Security` is best set at the nginx/HTTPS layer.

### SEC8 — Session fixation on privilege change — **FIXED** (was low)
Login/register reused the pre-auth session id. Fix: both now call
`req.session.regenerate()` before setting `userId`/`role`, so a new session id
is issued at the moment of authentication.

### SEC9 — Commenter email addresses exposed via API — **FIXED** (was low)
`GET /api/books/:slug` returned each commenter's raw `email` in the JSON (the
client only used it as a display fallback). Fix: the query no longer selects
`email`; it computes `display_name` server-side (falling back to the email's
local part), so addresses never leave the server. The book page template falls
back to "Neighbor".

### SEC10 — Open-redirect hardening on login `next` — **FIXED** (was low)
`login.html` redirected to a `next` query param after sign-in. Fix: it now
accepts only same-site paths (must start with `/`, not `//`), preventing an
attacker-crafted `?next=https://evil` link from bouncing an authenticated user
off-site.

**Already correct (OK):** parameterized SQL throughout (the only interpolated
identifiers are the fixed `APPROVABLE` map and `uniqueSlug`'s hardcoded table
names — not user-controlled); bcrypt cost 12; server-side SQLite sessions;
`httpOnly`/`secure`(prod)/`sameSite=lax` cookies (lax blocks the main CSRF
vector); `x-powered-by` disabled; 256kb body cap; admin routes gated with
`requireAdmin`; public detail pages escape all rendered fields; `rel="noopener"`
on outbound links; hardened deploy script + systemd unit (dropped capabilities,
`ProtectSystem=strict`, `PrivateTmp`, scoped `ReadWritePaths`, random
`SESSION_SECRET`/`ADMIN_PASSWORD`, refuses to clobber other apps' files).

## SEO

### S1 — Detail pages weren't individually indexable — **FIXED** (was medium)
`business.html`, `event.html`, `book.html`, `aid-post.html` are static shells
served for every slug, with generic hardcoded `<title>`/description and a
canonical pointing at the **list** page — telling Google the detail pages are
duplicates, and giving social scrapers (which don't run JS) only the generic
preview. Fix: the slug routes in `server.js` now read the DB row and rewrite the
`<head>` server-side — real title, description (from the item), a
self-referencing canonical, and `og:title`/`og:url`/`og:description` — before
sending HTML. The client JS still renders the body. Verified: `/directory/:slug`
returns the business's own title and canonical, and the injected values are
HTML-escaped. Unknown slugs now return a real **404** instead of a soft-404
shell.

### S2 — Ephemeral aid posts were indexable — **FIXED** (was medium)
Aid posts expire after 30 days but the detail page was indexable with a
canonical pointing at `/mutual-aid`. Fix: `aid-post.html` now carries
`<meta name="robots" content="noindex">` (consistent with its exclusion from
the sitemap).

### S3 — Sitemap had no `<lastmod>` — **FIXED** (was low)
Fix: `/sitemap.xml` now emits `<lastmod>` (from each row's `created_at`) for
business/event/book detail URLs.

### S4 — `robots.txt` hostname hardcoded — **FIXED** (was low)
The static `robots.txt` hardcoded `Sitemap: https://locallee.org/...` even
though the sitemap `loc`s are built from `SITE_URL`. Fix: `robots.txt` is now
served dynamically from `SITE_URL` (the static file was removed), so the two
always agree regardless of launch domain.

### S5 — Favicon missing — **FIXED** (was low)
Fix: added `public/favicon.svg` (the brand mark) and a
`<link rel="icon" type="image/svg+xml">` in the shared template and both
hand-authored pages.

### S6 — No `og:image` / `twitter:image` — **DEFERRED** (low)
All pages use `twitter:card = summary` with no image, so link previews are
text-only. Add a single default social image (1200×630) referenced site-wide;
best paired with a per-item image on detail pages later. Not done here (needs a
designed asset).

**Already correct (OK):** unique per-page `<title>`/`meta description`; one
`<h1>` per page; `lang="en"`; viewport; canonical on every page; OG + Twitter
tags; JSON-LD (`WebSite`+`Organization` on home; `LocalBusiness`/`Event`/`Book`
on detail pages); dynamic sitemap; `robots.txt` disallows `/admin` and `/api/`;
`noindex` on admin; semantic HTML with skip-link and ARIA attributes.

## Deferred (recommended follow-ups)

1. **SEC3 residual** — clear the build-time `tar`/`node-gyp` chain once
   `connect-sqlite3`/`sqlite3` ship a newer `node-gyp`, or move the session
   store off `sqlite3`.
2. **SEC7 tightening** — move inline page scripts to files and drop
   `'unsafe-inline'` from the CSP (switch to nonces).
3. **S6** — add a default `og:image`.
4. Add `Strict-Transport-Security` at the nginx layer once HTTPS is confirmed.
5. Consider nginx `limit_req` in front of the app as defense-in-depth for
   rate limiting.
